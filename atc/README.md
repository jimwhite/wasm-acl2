# wasm-vm1 — a WebAssembly interpreter built via ACL2 / ATC

This directory contains a tiny but real WebAssembly interpreter whose C
implementation is **generated from an ACL2 specification** by Kestrel's
**ATC** ("ACL2 To C") code generator. The end product, `./wasm-vm1`, loads a
`.wasm` file, resolves an exported function by name, and interprets its
body against a concrete operand stack and local slots.

At the M1 milestone it executes the subset of WASM needed to run the
`gcd` fixture and agrees with V8 on all oracle cases:

```
gcd a=48  b=18  ours=6   v8=6   OK
gcd a=35  b=14  ours=7   v8=7   OK
gcd a=17  b=0   ours=17  v8=17  OK
gcd a=0   b=5   ours=5   v8=5   OK
gcd a=1   b=1   ours=1   v8=1   OK
```

## 1. What is ATC, and what exactly is verified?

ATC is a verified C code generator. You write an ACL2 function in a
restricted style (the "ATC fragment": specific C-typed helpers like
`c::add-sint-sint`, `c::uchar-array-read`, `c::condexpr`, loops encoded
as tail-recursive `...$loop` functions, etc.). ATC then:

1. **Proves your ACL2 function admissible** — ACL2 checks termination and
   guard verification. Guard verification means: for every call of a
   typed operator (`add-sint-sint`, `uchar-array-read`, struct field
   accessor, array index read, ...) ATC has arranged that you must
   discharge a proof obligation showing the operation is well-defined
   (no signed overflow, no out-of-bounds array access, no division by
   zero, indexes fit the struct array length, etc.). If any of those
   proofs fail, you don't get C.

2. **Emits C code** (`wasm-vm1.c` / `wasm-vm1.h`) that is a syntactic
   translation of your ACL2 function under a formally-specified semantics
   of the C subset ATC supports (based on the Kestrel C formalization).

3. **Emits, as part of its trusted generator, a proof that the generated C,
   when executed under that C semantics, is behaviorally equivalent to
   the ACL2 function.** In the ATC paper's language: the generated C
   *refines* the ACL2 spec. We did not enable `:proofs t` on the `c::atc`
   call here (that option would also generate and discharge a per-function
   refinement theorem into the current world), so what we are relying on
   at this stage is:

   - ACL2 admits the spec (termination + guard verification proven).
   - ATC is a trusted translator: given an admitted spec in its fragment,
     the C it emits matches the spec under the ATC C semantics.

   Turning on `:proofs t` gives per-program, in-book theorems of the form
   "the C execution of `invoke` implements the ACL2 `|invoke|`". That is
   an available next step and is essentially a switch, not new work; it
   just increases certification time.

So the verification delivered today is:

- **Type-safety / memory-safety of the interpreter at the ACL2 level.**
  Guard verification has proved every array read/write, struct field
  access, signed arithmetic operation, and struct-array index is
  well-formed under the stated preconditions. These are the classic
  undefined-behavior classes of C; we have machine-checked proofs that
  they do not occur.

- **Totality.** Both recursive loops (`|scan$loop|` and `|exec$loop|`)
  are admitted with explicit `fuel` measures, so the interpreter is a
  total ACL2 function. It cannot diverge: on pathological input it
  exhausts fuel and returns.

- **Faithful translation.** ATC is a verified generator; the C code is
  a mechanical image of the ACL2 spec. There is no hand-written C to
  audit (except the 80-line `main.c` driver, which is clearly separated).

What is **not** yet proven in this book:

- Semantic equivalence to the official WebAssembly specification. That
  would be a separate theorem: "for programs `p` in our subset, our
  interpreter agrees with the WASM reference semantics." We instead
  *empirically* oracle-verify against V8.
- End-to-end C-level refinement (the `:proofs t` theorems described
  above are not yet emitted into the book).

Both are planned follow-ons.

## 3. Architecture

There are four layers, top to bottom:

```
  main.c (thin driver, ~80 lines of hand-C)
    │   reads file → copies bytes into global wasm_buf[65536]
    │   calls parse_module(&m), checks m.err, checks export name
    │   allocates struct wst st = {0}
    ▼
  invoke(&st, &m, a, b)            ← generated C from |invoke|
    │   st.loc[0] = a; st.loc[1] = b
    │   pc = clamp(m.body_off, 0, 60000)
    │   runs exec$loop(st, sp=0, nl=0, pc, halted=0, fuel=100000)
    │   returns st.op[0]  (the result WASM left on the stack)
    ▼
  exec$loop           ← generated C from |exec$loop|
    │   10-opcode dispatch, stack machine, label stack
    ▼
  wasm_buf[]          ← external uchar array; read-only after parse
```

### 3.1 The parser

Two generated C functions:

- `parse_module(struct wmod *m)` walks the section headers and fills in
  `wmod` — a small record describing the module.
- `parse$loop` is its tail-recursive worker.

`struct wmod` captures just what the interpreter needs:

```c
struct wmod {
    unsigned char err;           // nonzero if parse failed
    int body_off, body_len;      // where the function body lives in wasm_buf
    unsigned char num_params;
    unsigned char num_results;
    unsigned char num_locals;
    int export_off;              // offset of the first export name
    unsigned char export_len;    // its length
};
```

The parser is **generic**: it walks the standard section IDs (type, func,
export, code), reads LEB128 small-int fields, and records offsets. It
makes no assumption about the function's name or body; the word "gcd"
does not appear anywhere in `wasm-vm1.lisp`. The export gate in `main.c`
simply compares `wasm_buf[export_off .. export_off+export_len]` against
`argv[2]`.

### 3.2 The interpreter state

```c
struct wst {
    unsigned int  op[64];      // operand stack
    unsigned int  loc[16];     // locals (params + declared locals)
    int           lpc[16];     // label-stack: jump PCs
    int           lsp[16];     // label-stack: operand-stack heights
    unsigned char lkind[16];   // label-stack: 0 = block, 1 = loop
};
```

Plus five scalars passed through `exec$loop` as separate arguments
(not struct fields) so their ranges can be carried as guard invariants:
`sp ∈ [0,64]`, `nl ∈ [0,16]` (label depth), `pc ∈ [0,60000]`,
`halted ∈ {0,1}`, `fuel ∈ [0,100000]`.

### 3.3 The dispatch loop

`|exec$loop|` is a nested cascade of `if`s, one per opcode. Because ATC
requires each recursive call to be on the original formals, every arm
ends with a `(|exec$loop| st sp' nl' pc' halted' fuel' wasm_buf)`.

Supported opcodes (WASM 1.0 subset sufficient for gcd):

| Hex  | Opcode        | Effect                                              |
|------|---------------|-----------------------------------------------------|
| 0x20 | local.get X   | push `loc[X]`                                       |
| 0x21 | local.set X   | `loc[X] = pop()`                                    |
| 0x22 | local.tee X   | `loc[X] = peek()`                                   |
| 0x45 | i32.eqz       | push `top == 0`                                     |
| 0x70 | i32.rem\_u    | `rhs = pop(); lhs = pop(); push(lhs % rhs)`         |
| 0x02 | block BT      | push a block label; target pc = after matching `end`|
| 0x03 | loop BT       | push a loop label;  target pc = this loop header    |
| 0x0b | end           | pop label, or halt if at function end               |
| 0x0c | br L          | pop to label L's sp, jump to its target pc          |
| 0x0d | br_if L       | if `pop()` nonzero, do `br L`; else fall through    |

Each opcode arm uses `c::condexpr` to express the C `?:` operator for
pure-expression conditional values (e.g. clamping or branch target
selection) and `c::declar` / `c::assign` to mark local-variable
introductions and updates in the ATC fragment.

### 3.4 The `|scan$loop|` helper (forward end matcher)

`block`, `loop`, and `br/br_if` need the address of the matching `end`
(or the next-instruction address). Rather than pre-compute a jump table,
the interpreter *scans forward* from the start of a block with a small
depth counter and a fuel budget of 4096 steps per scan. When a `br`
targets a `block` we use the scanned end-pc as the branch target;
when it targets a `loop` we use the loop header pc. This keeps the
interpreter memoryless between invocations — no precomputed side
tables — and preserves easy termination arguments.

### 3.5 Why the scalars are "threaded", and why the state is a struct

ATC's surface rules are narrow about `mv`-return:

- Inside a `*$loop` function, multi-value return is fine — ATC inlines
  it as a `while` loop in the caller, with each returned scalar stored
  into its matching local variable.
- Outside a loop, ATC will emit one scalar return by value; pointers /
  structures / external objects are emitted as by-reference parameters.

That's exactly why the interpreter keeps arrays inside `struct wst`
(passed as a pointer and mutated in place) and keeps the hot scalars
(`sp`, `nl`, `pc`, `halted`, `fuel`) as separate parameters threaded
through the tail-recursive loop. The generated C ends up as idiomatic
stack-machine code: a `while` over the opcode table, mutating five
local ints and the state through a pointer.

## 4. Anatomy of an opcode arm

A representative arm — `local.get X` — looks like this in the ACL2
source:

```lisp
(if (c::boolean-from-sint (c::eq-sint-sint |b| (c::sint-dec-const #x20)))
    (let* ((|x|   (c::declar (byte-at (c::add-sint-sint
                                        |pc| (c::sint-dec-const 1)))))
           (|ok|  (c::declar
                   (c::sint-from-boolean
                    (and (c::boolean-from-sint
                          (c::lt-sint-sint |x| (c::sint-dec-const 16)))
                         (c::boolean-from-sint
                          (c::lt-sint-sint |sp| (c::sint-dec-const 64)))))))
           ;; read loc[x] (clamp to 0 if not ok)
           (|v|   (c::declar
                   (c::condexpr
                    (if (c::boolean-from-sint |ok|)
                        (struct-|wst|-read-|loc|-element |x_clamped| |st|)
                      (c::uint-dec-const 0)))))
           (|st|  (struct-|wst|-write-|op|-element |sp| |v| |st|))
           (|sp|  (c::assign (c::add-sint-sint |sp| (c::sint-dec-const 1))))
           (|pc|  (c::assign (c::add-sint-sint |pc| (c::sint-dec-const 2))))
           (|fuel| (c::assign (c::sub-sint-sint |fuel| (c::sint-dec-const 1)))))
      (|exec$loop| |st| |sp| |nl| |pc| |halted| |fuel| |wasm_buf|))
  ... next opcode ...)
```

Points worth noting:

- `c::boolean-from-sint` / `c::sint-from-boolean` marshal between the
  ACL2 propositional world and the ATC-typed `sint` that matches C's
  `int`-valued conditions.
- `c::condexpr` is the ATC marker for "this `if` must be emitted as a C
  `?:` because it appears in expression position" (e.g. inside a
  `c::assign`, which is pure).
- `struct-|wst|-write-|op|-element` is a generated **functional**
  writer that returns a new `wst`. The surrounding `let*` rebinds
  `|st|`, which ATC emits as a mutation through the `wst *` parameter
  in the C.
- Every opcode arm decrements `fuel`; the loop halts when fuel hits
  zero, which is how `|exec$loop|` is shown to terminate.

## 5. What verification we get, concretely

At `:proofs nil` (today), certification gives us:

- **Termination.** `|scan$loop|` terminates on a fuel measure; so does
  `|exec$loop|`. Hence both are total ACL2 functions. No inputs make
  the spec (or therefore the emitted C) loop forever.
- **Guard verification** for every typed operator in the fragment. In
  ATC terms this rules out the C-level undefined behaviors that
  correspond to those operators:
    - Every `c::uchar-array-read` / `uchar-array-write` on `wasm_buf`
      or on a struct array has its index proven in range.
    - Every signed-int arithmetic op (`add-sint-sint`, `sub-sint-sint`)
      has its no-overflow side condition discharged (this is why you
      see `sint-max->=-65600` and the various bound lemmas).
    - Every struct-array element access has its `|STRUCT-wst-X-INDEX-OKP|`
      proof.
    - Every conversion (`sint-from-uchar`, etc.) has its okp proof.
  Concretely: the interpreter cannot read or write out of `wasm_buf`,
  out of `st->op` / `loc` / `lpc` / `lsp` / `lkind`, cannot overflow
  `int` arithmetic, cannot divide by zero (for `rem_u` we explicitly
  branch on the zero divisor), and cannot dereference a NULL struct.
- **Faithful C.** ATC's translator correctness means the emitted C
  implements the admitted ACL2 function under ATC's C semantics.

What is *not* part of the book:

- **Agreement with the WASM reference semantics.** The oracle tests are
  our empirical check against V8; a formal refinement to a WASM model is
  future work.
- **End-to-end C theorems** (`:proofs t`). Not emitted yet — switching
  on that flag would add them; it does not require spec changes.
- **The driver.** `main.c` is ordinary C. It is small, but it is not
  verified.

## 8. Roadmap

Short-term (M2+):

- Turn on `:proofs t` in the `c::atc` call to emit per-function
  C-refinement theorems into the book.
- Extend opcode coverage (arithmetic, comparisons, `if/else`, `return`,
  `call`) until we can run the `sum-spec`/`call-indirect-spec` fixtures
  under `proofs/`.
- Start a formal WASM-subset semantics in ACL2 and prove
  `|exec$loop|` refines it. That gives a machine-checked bridge from
  "it agrees with V8 empirically" to "it agrees with the spec".

Medium-term:

- Stop hard-coding stack/local/label bounds and parameterize the
  struct sizes.
- Replace the 65 KB `wasm_buf` external object with a proper pointer +
  length module handle once ATC's module-handle support lands on our
  branch.
