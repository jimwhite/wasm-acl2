# ATC → Verified C WASM Interpreter — Design Plan (rev 2)

**Goal.** Use Kestrel's ATC (ACL2-to-C) to emit a C implementation of a
WASM interpreter whose correctness against `execution.lisp` is proved
by construction, and whose executable can run the `.wat`/`.wasm`
programs already in [tests/oracle/](tests/oracle/) — producing the
same values the V8 oracle produces.

**Revisions in rev 2:**
- We are **not** writing a new binary decoder. The Kestrel WASM books
  already ship `books/kestrel/wasm/parse-binary.lisp` (1429 LOC, LEB128,
  full WASM 1.0 binary format) and we already depend on it.
- **Every milestone ends with "compile the C, run it, diff against
  V8."** No preparatory work lands without a runnable artifact that
  demonstrates it was worth doing.

---

## 1. ATC — what it supports and doesn't

Confirmed from `books/kestrel/c/atc/`:

### Supports
- Integer types (`schar` … `ullong`) with tight guards, overflow, all
  bitwise/shift ops.
- Structs (`defstruct`), arrays, pointers, pointed integers.
- Loops via tail recursion; conditionals; local variables; function
  calls; external objects (`defobject`).

### Does not support
- **Floating point.** `pretty-printer.lisp` raises on float constants.
  Blocks f32/f64 programs until upstream ATC grows float emission.
- General recursion, symbols, quasiquote, alists, dynamic allocation,
  unions, variadics.

This forces the WASM interpreter used for ATC to be a **second
implementation** in the ATC-friendly subset, linked to the spec in
`execution.lisp` by a refinement proof (milestone M6 below).

---

## 2. Scope cuts for v0

1. **Integer-only subset of oracle programs:** `gcd`, `sum_array`,
   `hash`, `collatz`, `is_prime`, `packed_mem`, `factorial`,
   `call_indirect`, `edge_cases` (non-float parts). Floats deferred
   until ATC upstream grows float support (milestone M8).
2. Single module, single instance; no host imports; interpreter only;
   all stacks/memory bounded at compile time.

---

## 3. Architecture

```
┌────────────────────────────────────────────────┐
│ execution.lisp            (unchanged)          │  L0  — spec
└────────────────────────────────────────────────┘
        ▲
        │ refinement theorem (milestone M6)
        │
┌────────────────────────────────────────────────┐
│ execution-atc.lisp                             │  L1  — ATC mirror
│   - struct-of-arrays state                     │
│   - enumerated opcodes                         │
│   - tail-recursive step-atc                    │
└────────────────────────────────────────────────┘
        ▲  emits
┌────────────────────────────────────────────────┐
│ wasm_vm.c  +  wasm_vm.h    (generated by ATC)  │  L2
│ + ACL2 theorems that wasm_vm.c == step-atc     │
└────────────────────────────────────────────────┘

Glue (lives outside ATC):

  parse-binary.lisp   kestrel/wasm   — already in-tree WASM binary
                                       parser (1429 LOC). We CALL it,
                                       not rewrite it.

  module-lower.lisp   new            — runs at build time in ACL2:
                                       calls parse-binary on a .wasm,
                                       lowers the result to our flat
                                       struct-of-arrays form, and
                                       emits a C header containing
                                       the module as initialized
                                       struct literals.

  main.c              new, ~200 LOC  — reads compiled-in module,
                                       parses argv, calls
                                       wasm_vm_invoke, prints result.
```

**Key move.** Binary decoding happens in ACL2 *at build time* using the
Kestrel parser. The VM itself ships each program as a C header of
initialized struct literals. No runtime decoder for milestones M1–M4;
runtime loading is added in M5 as a thin wrapper that fills the same
struct shape. This is the pattern Kestrel's own ATC examples use.

**Trust base for M4:**
- ATC itself
- Our C compiler
- `main.c` (trivial CLI)
- The build script that runs `parse-binary` + `module-lower` offline
  and writes the header

`execution.lisp` + refinement proof (M6) reduce `wasm_vm.c` to the
spec. `parse-binary.lisp` is already a formal ACL2 artifact; its
verification quality is upstream's problem, and we can tighten our
trust story further in M7 by ATC-compiling it too.

## 4. Milestones — each ends with a runnable artifact

### M1. Smallest WASM VM in C (via ATC) that runs `gcd.wasm` — ~2 weeks

**What.** A minimal but real WASM interpreter, generated by ATC from
ACL2.

- `execution-atc.lisp` v0: the ATC-friendly interpreter.
  - `wasm-state` defstruct: operand stack, label stack, frames, locals
    (all fixed-size arrays; no memory yet).
  - `wasm-module` defstruct: opcode array + immediates + function
    entries.
  - Enumerated opcodes — the 9 needed by `gcd.wat`:
    `local.get`, `local.set`, `block`, `loop`, `br`, `br_if`,
    `i32.eqz`, `i32.rem_u`, `end`.
  - Tail-recursive `step-atc` dispatching on opcode.
  - Top-level `wasm-vm1-invoke(module, fn-idx, a, b) → uint`.
- ATC invocation emits **`wasm-vm1.c`** + `wasm-vm1.h`.
- Hand-written `module_gcd.h`: `gcd.wat` encoded directly as
  struct-literal instruction array (NO parser involvement — that
  starts in M3). ~25 lines.
- `main.c` (~100 LOC): parses argv, calls `wasm_vm1_invoke`, prints.

**Demo.**
```
$ make wasm-vm1-gcd
$ ./wasm-vm1-gcd 48 18        # => 6
$ ./wasm-vm1-gcd 35 14        # => 7
$ ./wasm-vm1-gcd 17  0        # => 17
$ make oracle-verified-m1     # diffs every gcd oracle case vs V8
PASS  gcd(48,18)   vm=6   v8=6
PASS  gcd(35,14)   vm=7   v8=7
PASS  gcd(1,1)     vm=1   v8=1
PASS  gcd(17,0)    vm=17  v8=17
PASS  gcd(0,5)     vm=5   v8=5
```

**Exit criterion.** `wasm-vm1.c` is ATC-generated. `./wasm-vm1-gcd`
matches V8 on all five gcd oracle cases. **If ATC can't produce a
working `wasm-vm1.c` from this scope, stop** and consult Kestrel before
further work.

**Risk/probe combined.** This is the real feasibility check: can ATC
emit a dispatch-on-enum tail-recursive interpreter with a
moderately-sized state struct? The WASM VM shape is present from the
start; everything after M1 is extension.

### M2. Factorial: hand-curated — ~1 week

**What.** Exercise the same VM on a second program before touching the
parser, to confirm the generalization holds.

- Add opcodes needed by `factorial.wat`: `i32.const`, `i32.mul`,
  `i32.sub`. (The others are already present from M1.)
- Hand-written `module_factorial.h` in the same struct-literal style.
- Reuse `wasm-vm1.c` unchanged if possible; if `step-atc` needs new
  opcode branches, regenerate.

**Demo.**
```
$ make wasm-vm1-factorial
$ for n in 0 1 2 5 10 12; do
    printf "n=%s  vm=%s  v8=%s\n" \
      "$n" "$(./wasm-vm1-factorial $n)" "$(oracle value)"
  done
# all equal
```

**Exit criterion.** Same VM binary runs both `gcd` and `factorial`
modules, matching V8 on both oracle tables.

### M3. Plug in the Kestrel parser; `module-lower.lisp` — ~1.5 weeks

**What.** Replace the hand-written `module_*.h` with build-time
generation.

- `module-lower.lisp`: an ACL2 script that
  (a) calls `parse-binary` on a given `.wasm`,
  (b) walks the parser's structured output,
  (c) emits `module_<name>.h` as struct literals matching what M1/M2
      wrote by hand.
- Regenerate `module_gcd.h` and `module_factorial.h` from the real
  `.wasm` files; confirm they're byte-equivalent (or behaviorally
  equivalent) to the hand-curated versions.

**Demo.**
```
$ make module_gcd.h           # runs ACL2 parser + lowering on gcd.wasm
$ make module_factorial.h     # same for factorial.wasm
$ make oracle-verified-m3     # rebuilds everything from .wasm, diffs vs V8
# identical pass output to M2
```

**Exit criterion.** Headers are generated from real `.wasm` files, not
hand-written. Oracle diff still green for gcd and factorial.

### M4. Full integer WASM 1.0, full integer oracle passing — ~4 weeks

**What.**
- Remaining i32/i64 opcodes: shifts, rotates, comparisons, conversions,
  load/store (8/16/32-bit, signed/unsigned), `br_table`, `call`,
  `call_indirect`, `return`, `select`, `drop`, `local.tee`, globals.
- Linear memory added to `wasm-state`.
- `module-lower.lisp` extended to cover all these forms.

**Demo.**
```
$ make oracle-verified
PASS  gcd(48,18)=6              vm=6        v8=6
PASS  sum_array(0,5)=150         vm=150      v8=150
PASS  hash("Hi")=5862390         vm=5862390  v8=5862390
PASS  collatz(27)=111            vm=111      v8=111
PASS  is_prime(97)=1             vm=1        v8=1
PASS  factorial(10)=3628800      vm=3628800  v8=3628800
PASS  packed_mem.load8_u(0)=171  vm=171      v8=171
PASS  call_indirect(…)           vm=…        v8=…
```

**Exit criterion.** `make oracle-verified` passes in CI. Diffs V8 vs our
VM for every integer `.wat` and exits non-zero on any mismatch.

### M5. Runtime `.wasm` loading — ~1 week

**What.** Until now each program had its module baked in. Now make the
VM generic: a small `load_module.c` reads a `.wasm` at runtime and fills
the struct arrays in the shape `module-lower.lisp` produces offline.

This loader is trusted; the alternative (ATC-compiling the full
parser) is M7.

**Demo.**
```
$ ./wasm_vm tests/oracle/factorial.wasm fac 10
3628800
$ ./wasm_vm <any-integer-only.wasm> <fn> <args...>
```

**Exit criterion.** One VM binary, handed any integer `.wasm` in the
oracle set, matches V8.

### M6. Refinement proof: `execution-atc` refines `execution.lisp` — ~8 weeks, overlappable

**What.**
- `match-state` abstraction from flat struct → tree-of-alists spec
  state.
- Per-opcode simulation lemmas (~60).
- Structural lemmas (operand/label/frame stack ops, IP advance, branch
  target resolution).
- Main theorem:
  `(match-state s-atc s-spec) ⇒ (match-state (step-atc s-atc m) (run s-spec))`
  extended by fuel induction.

This milestone **runs in parallel with M4 and M5** once M2 has
stabilized the state struct. Each new opcode in M3/M4 comes with its
per-opcode lemma in M6 before the opcode is called "done."

**Demo.** `make oracle-verified` stays green throughout. New
`make refinement-proof` target certifies the ACL2 books.

**Exit criterion.** Main refinement theorem certifies; no
`skip-proofs`; oracle suite still green.

### M7. (Optional) ATC-compiled parser — ~4 weeks

**What.** Compile `parse-binary.lisp` through ATC, producing
`parse_binary.c`. Replace the trusted `load_module.c`. Trust base
shrinks; decoder correctness follows from ATC + the Kestrel parser's
existing verification.

**Demo.** Same behavior as M5, but `./wasm_vm` now uses the generated
decoder.

### M8. (Blocked upstream) Floats

**What.** When ATC grows f32/f64 emission, add float state + opcodes;
rerun the full oracle including `float_ops`, `nan_oracle`,
`inf_arith_oracle`, `signed_zero_oracle`.

**Fallback if upstream ATC doesn't ship floats:** hand-write
`wasm_floats.c` (~500 LOC), test it against V8 via a fuzzer (not
formal proof), link it in. Weakens trust for float programs only.

### M9. (Future direction) WASM-to-C AOT compiler with ACL2 models — open-ended

**What.** The milestones above build a *verified interpreter*: one
fixed C program (`wasm-vm.c`) that dispatches on opcodes at runtime.
This milestone flips the pipeline: instead of interpreting each module,
**compile** each WASM module *ahead of time* into a bespoke C program
(or other target — Rust, LLVM IR, native) whose control flow is the
module's control flow and whose arithmetic is direct C arithmetic. No
opcode dispatch loop at runtime. Expected speedup over M4/M5: 1–2
orders of magnitude, approaching V8's baseline tier.

**Pipeline shape.**

```
   .wasm
     │
     ▼  parse-binary.lisp                  (already in-tree)
   structured module
     │
     ▼  wasm-to-acl2.lisp                  (new: lift to ACL2 funcs)
   ACL2 defun per WASM function, in the ATC-friendly subset
     │
     ▼  ATC                                 (existing)
   module_foo.c  (one C function per WASM function, no interpreter)
     │
     ▼  cc
   native binary
```

The key insight the M-chain above unlocks: once `execution-atc.lisp`
has normalized operand-stack / label-stack / locals into a flat
representation, each WASM function is already a tail-recursive ACL2
function modulo a lifting pass. Loops lift to tail-recursive helpers
(we already have prototypes in `wasm-loop-lifter.lisp`); straight-line
code lifts to `let*` chains; `call` lifts to ACL2 function call; `br`
lifts to `return` from the enclosing helper. ATC already knows how to
emit all of this.

**ACL2 model per program.** Because the lifter produces ACL2 defuns,
each compiled program ships with:
  - its C source (fast execution),
  - its ACL2 model (the same defun, executable and reasoning-friendly),
  - a proof obligation: `(equal (compiled-foo args) (run-wasm foo
    args))`, discharged by the same refinement strategy as M6 but
    per-function instead of over the whole VM.

So M9 gives verified AOT compilation: every WASM program becomes
a C program *plus* a machine-checkable proof that the C program
computes what the WASM program computes.

**Dependencies.**
- M4 (all integer opcodes understood in ACL2).
- M6 (refinement theorem, so the ACL2-level semantics we compile
  *from* is pinned to the spec).
- Loop-lifter research (existing `WASM_LOOP_LIFTER_PLAN.md`).

**Non-goals for M9.**
- Beating V8's optimizing tier. Baseline JIT is the target.
- JIT / dynamic recompilation. This is pure AOT.

**Why note it now.** The interpreter work (M1–M7) produces exactly the
infrastructure M9 needs — parser, ACL2 state model, refinement
lemmas, ATC pipeline. Keeping M9 in view lets us avoid architectural
choices in M2–M4 that would wall it off (e.g. baking opcode dispatch
assumptions into state shape).

## 7. Risks

- **R1 — ATC can't emit the shape we need.** M1 is the probe. Unlike
  the earlier plan, M1 is itself a minimal WASM VM, so success means
  the fundamental approach works; failure means stop before M2. We
  don't pay for a separate throw-away probe.
- **R2 — `step-atc` too big to emit as one C function.** Split into
  per-opcode ACL2 functions → per-opcode C functions. Larger refinement
  proof but cleaner C. Detected in M1 (9 opcodes already exercises
  dispatch).
- **R3 — Kestrel parser's output shape awkward for lowering.**
  Detected in M3. Mitigated by doing lowering in ACL2.
- **R4 — Float gap.** Known; M8.
- **R5 — Performance.** Verified interpreters are typically 10–100×
  slower than V8; expected and acceptable.

## 8. Recommendation

Do M1 this week-ish. One ACL2 file defining a 9-opcode WASM
interpreter, one ATC invocation producing `wasm-vm1.c`, one
hand-written `module_gcd.h`, one `main.c`, one `./wasm-vm1-gcd`
binary, one diff against V8. If that lands, we have proof the WASM-VM-
via-ATC pipeline works on a real (small) WASM program, and the rest is
engineering driven by a runnable artifact at every step.
